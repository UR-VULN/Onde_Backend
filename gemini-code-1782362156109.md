## 3. 총평
[cite_start]진단 결과 게시판 내 **UNION 기반 SQL Injection**을 통한 전체 회원 정보 탈취, 회원가입 시 파라미터 조작을 통한 **SUPER_ADMIN 권한 상승**, **항공권 결제 금액 위변조**, 정산서 PDF 출력 기능에서의 **LFI 및 SSRF를 통한 AWS 인스턴스 자격증명(IAM Role Key) 탈취** 등 시스템 인프라 전체를 무력화할 수 있는 **심각한(Critical/High) 취약점들이 다수 발견**되었습니다. [cite: 48, 61] [cite_start]조치 우선순위에 따라 긴급 보안 대책 수립 및 패치가 강력히 권고됩니다. [cite: 50]

## 4. 취약점 진단 상세 결과

### 4.1 입/출력 값 검증 부재

#### [1-1] XSS / CSRF 공격 가능성 (중요)
* [cite_start]**상세 내용:** 게시글 작성 기능(`POST /user-api/api/v1/posts`)의 `title`, `content` 필드에 서버 측 필터링이 전혀 없어 Stored XSS 페이로드(`%22%3E%3Cimg src=x onerror=alert(1)%3E`)가 DB에 그대로 저장됨을 확인했습니다. [cite: 58] [cite_start]프론트엔드 React JSX의 기본 Escape 처리로 우연히 방어되고 있으나, `dangerouslySetInnerHTML`을 사용하는 컴포넌트 배치 시 즉시 스크립트가 실행될 위험이 존재합니다. [cite: 58]
* [cite_start]**대응 방안:** 1. 서버 측 입력 단계에서 OWASP Java HTML Sanitizer 등을 이용해 시큐어 코딩(Sanitizing) 적용 [cite: 58]
  2. [cite_start]출력 시 HTML 이스케이프 처리 [cite: 58]
  3. [cite_start]CSP(Content-Security-Policy) 헤더 도입 [cite: 58]

#### [1-2] 삽입 (Injection) 공격 가능성 (중요)
* [cite_start]**상세 내용:** 게시글 검색 API(`GET /user-api/api/v1/posts?status=`)의 `status` 파라미터에서 **UNION 기반 SQL Injection** 결함 발견. [cite: 59] [cite_start]일반 사용자 권한으로 `members` 테이블을 조회하여 **전체 회원의 이메일과 bcrypt 비밀번호 해시를 평문으로 탈취 가능**한 매우 심각한(Critical) 상태입니다. [cite: 59] (*참고: keyword 파라미터는 PreparedStatement가 적용되어 안전함*) [cite_start][cite: 59]
* **대응 방안:** 동적 쿼리를 지양하고 PreparedStatement/JPA 파라미터 바인딩 일괄 적용. [cite_start]`status` 파라미터는 Java Enum 타입으로 처리하여 예외 값 유입 시 즉시 400 에러 처리. [cite: 59]

#### [1-3] 파라미터 값 및 히든 필드 조작 가능성 (중요)
* [cite_start]**상세 내용:** 회원가입(`POST /user-api/api/v1/auth/signup`) 및 역할 전환 API 요청 Body에 클라이언트가 임의로 `"role":"SUPER_ADMIN"` 주입 시 서버 측 검증 없이 최고 관리자 권한 계정이 생성되거나 승격되는 현상 발생. [cite: 60] [cite_start]가입 후 관리자 전용 API 접근 및 전체 회원 정보 유출로 직결됩니다. [cite: 60]
* **대응 방안:** 회원가입 시 클라이언트 입력과 관계없이 `USER` 권한으로 고정 처리. 역할 전환 역시 `SELLER` 외 권한 변경 프로세스는 별도의 안전한 승인 절차로 완전 분리. [cite: 60]

#### [1-4] SSRF / File Inclusion 공격 가능성 (중요)
* [cite_start]**상세 내용:** 마이페이지 정산서 생성 API(`POST /user-api/api/v1/report/integrated`)의 `template` 및 `logoUrl` 파라미터를 통해 **LFI(Local File Inclusion) 및 SSRF 공격 성공(Critical)**. [cite: 61]
  * [cite_start]**LFI:** `template`에 `../../../proc/self/environ` 주입 시 PDF 결과물 내에 **시스템 전역 환경변수(DB 패스워드, JWT_SECRET, OAuth 시크릿 키 등)가 평문 노출**됩니다. [cite: 61]
  * [cite_start]**SSRF:** `logoUrl`에 AWS 메타데이터 엔드포인트(`http://169.254.169.254/...`)를 주입하여 **EC2에 부여된 IAM 임시 보안 자격 증명(STS Token)까지 탈취가 확정**되었습니다. [cite: 61]
* **대응 방안:** * **즉시 조치:** 노출된 모든 시크릿 키 평문 즉시 로테이션(교체). EC2 인스턴스의 IMDS 설정을 **IMDSv2(Token 필수)**로 즉시 강제 전환. [cite: 61]
  * **구조 개선:** `template` 인자는 Enum 화이트리스트로 제한하고 외부 입력이 경로 조합에 사용되지 않도록 전면 재설계. [cite_start]시크릿 정보는 AWS Secrets Manager 등으로 관리 전환. [cite: 61]

#### [1-6] 입력 값 크기 및 무결성 검증 오류 (중요)
* [cite_start]**상세 내용:** 항공권 예약 요청(`POST /user-api/api/v1/flights/booking`) 시 결제 금액 파라미터 `totalPrice`를 **'1'로 변조하여 전송했으나, 서버에서 정상 수용하여 비즈니스석 예약 선점에 성공**함. [cite: 63] (반면 렌터카 및 숙소 객실 결제 로직은 DB 기반 재검증이 정상 작동하여 안전함) [cite_start][cite: 63]
* [cite_start]**대응 방안:** 항공권 예약 로직에도 클라이언트 전달 금액을 무시하고, `scheduleId` 기준 DB 실제 단가를 재조회하여 검증하는 로직 추가 적용. [cite: 63]

### 4.2 취약한 파일처리

#### [2-1] 악성코드파일 업로드 (중요)
* [cite_start]**상세 내용:** 게시글 이미지 업로드 시 파일명 확장자 및 매직바이트를 검증하지 않아 `hacker.php` 같은 웹쉘 파일이 파일명 변조만으로 S3/CloudFront 스토리지에 그대로 저장됩니다. [cite: 66] [cite_start]또한 숙소 등록 시 스크립트가 포함된 `thumb.svg` 파일 업로드가 허용되어 Stored XSS 우회 경로로 악용될 우려가 있습니다. [cite: 66]
* **대응 방안:** 서버 측 파일 확장자 화이트리스트 검증 및 Magic Byte 검증 필수 적용. [cite_start]파일 저장 시 랜덤한 파일명으로 재지정하고, SVG 업로드는 지양하거나 위험 태그 필터링 처리. [cite: 66]

### 4.3 취약한 접근 통제 관리

#### [3-2] 인증 실패 횟수 제한 (일반)
* [cite_start]**상세 내용:** 로그인 엔드포인트(`POST /user-api/api/v1/auth/login`) 및 관리자 로그인 시 무차별 대입(Brute Force) 공격에 대한 방어책(계정 잠금, Rate Limiting, CAPTCHA 등)이 전무하여 무제한 인증 시도가 가능합니다. [cite: 71]
* [cite_start]**대응 방안:** 로그인 5회 실패 시 계정 임시 잠금 기능 구현, `429 Too Many Requests`를 통한 Rate Limiting 구현. [cite: 71]

#### [3-3] 계정 정보 파악 가능성 (일반)
* [cite_start]**상세 내용:** 로그인 화면 문구는 안전하나, 회원가입 시 이메일/닉네임 중복확인 API(`GET /api/v1/auth/check-email`)에서 가입 여부에 따라 상이한 응답 값을 반환하여 시스템 내 가입된 계정 조사가 가능(User Enumeration). [cite: 72]
* [cite_start]**대응 방안:** 중복확인 API 문구를 중립적인 문구로 변경하거나 해당 엔드포인트에 타이트한 Rate Limiting 적용. [cite: 72]

#### [3-4] 관리자 페이지 분리 여부 (일반)
* [cite_start]**상세 내용:** 일반 사용자(`www.`)와 관리자 시스템(`admin.onde.click`)의 도메인 및 서버 분리는 만족하나, 도메인명에 직관적인 키워드(`admin`)가 포함되어 경로 유추가 매우 쉽습니다. [cite: 73]
* [cite_start]**대응 방안:** 관리자 전용 서브도메인 명칭을 무작위 난수형태나 비정형 단어로 교체하고, 인프라 단에서 관리자 IP 화이트리스트 접근 제어 결합 권고. [cite: 73]

#### [3-5] 검색엔진 정보 노출 가능성 (일반)
* [cite_start]**상세 내용:** `robots.txt` 설정이 부재하여 크롤러에 의한 민감 정보 인덱싱 제어가 불가능합니다. [cite: 74]
* [cite_start]**대응 방안:** 루트 디렉토리에 `robots.txt`를 생성하고 `/api/`, `/admin/` 등 핵심 경로에 대한 `Disallow` 설정 명시. [cite: 74]

### 4.4 취약한 인증 및 세션 관리

#### [4-1] 쿠키 및 웹 스토리지 조작 가능성 (일반)
* [cite_start]**상세 내용:** 발급되는 JWT 쿠키(`accessToken`, `refreshToken`)에 `Secure`, `HttpOnly` 속성은 정상 부여되었으나, `SameSite=None`으로 설정되어 있어 크로스 사이트 요청을 통한 CSRF 유도 공격에 노출되어 있습니다. [cite: 78]
* [cite_start]**대응 방안:** 쿠키의 `SameSite` 속성을 비즈니스 요건에 맞춰 `Lax` 또는 `Strict`로 변경 권고. [cite: 78]

#### [4-2] 인증(세션 및 토큰) 값 안전성 설정 여부 (일반)
* **상세 내용:** JWT 디코딩 시 Payload에 사용자 이메일, 역할(`role`) 정보가 평문 노출되고 있습니다. [cite_start]단, 서명(`alg: none` 변조 등) 위조 검증 및 중복 로그인 시 구(Old) 리프레시 토큰에 대한 서버 측 무효화 검증 구조는 안전하게 구축되어 있음을 확정했습니다. [cite: 79]
* [cite_start]**대응 방안:** JWT Payload에 이메일 대신 유추할 수 없는 `memberId` 내부 식별자만 포함하도록 데이터 최소화. [cite: 79]

#### [4-5] 일반계정 권한 상승 가능성 (일반)
* **상세 내용:** 수평적 권한 상승(BOLA/IDOR) 결함 확인. [cite_start]본인 인증 토큰으로 로그인된 상태에서 객실 예약 생성 시 Request Body의 `memberId`를 타인의 ID로 변조하면 **타인 명의로 예약이 정상 생성되는 취약점**이 확정되었습니다. [cite: 82] [cite_start]예약 번호가 순차 정수형태라 무차별 취약점 대입 공격 시 결제/정산 책임 소재의 심각한 교란을 야기할 수 있습니다. [cite: 82]
* [cite_start]**대응 방안:** API 파라미터에서 클라이언트가 전달하는 `memberId` 입력 항목을 전면 배제하고, 시큐리티 세션 컨텍스트(JWT 토큰의 sub 클레임) 내의 인증 정보에서 서버가 직접 추출해 할당하도록 로직 수정. [cite: 82]

### 4.5 중요 정보 저장/전송 처리 미흡

#### [5-2] 요청 및 응답 값 내 주요정보 포함여부 확인 (중요)
* [cite_start]**상세 내용:** 마이페이지 예약 조회 API 호출 시 응답 결과 데이터 스키마 내에 **승객의 실제 이름, 예약 코드 등이 전혀 마스킹 처리되지 않은 평문 상태로 전송**되고 있습니다. [cite: 86]
* [cite_start]**대응 방안:** 개인정보 및 민감 데이터 필드는 프론트엔드가 아닌 서버 레이어에서 마스킹 로직(예: 고유 코드 및 이름의 일부를 `*` 처리) 적용 후 응답 처리. [cite: 86]

### 4.6 부적절한 오류 처리

#### [6-1] 오류 페이지를 통한 정보 노출 여부 (일반)
* [cite_start]**상세 내용:** 예외 처리 핸들러(`GlobalExceptionHandler`) 구조적 결함으로 인해 존재하지 않는 경로 접근 등 4xx 에러 유형이 유입될 때 내부 프레임워크 클래스 경로 정보가 `systemMessage`에 포함되어 500 에러로 전출되는 현상 확인. [cite: 89] (`NoResourceFoundException` 등 내부 스택 정보 유출) [cite_start][cite: 89]
* [cite_start]**대응 방안:** 프로덕션 환경의 에러 응답 객체에서 상세 스택 정보 필드(`systemMessage`) 표출을 제거하고 `application.yml` 파일 내 `server.error.include-stacktrace=never` 옵션 활성화. [cite: 89]

### 4.7 취약한 컴포넌트 구성요소

#### [7-3] 서버 헤더정보 노출 (일반)
* [cite_start]**상세 내용:** 모든 API 응답 공통으로 HTTP 헤더에 `Server: nginx/1.31.1`과 같이 사용 중인 웹서버의 명확한 버전 인프라 구조 정보가 평문 표출되고 있습니다. [cite: 95]
* [cite_start]**대응 방안:** 백엔드 Nginx 설정 파일(`nginx.conf`)의 http 블록 내부 레이어에 `server_tokens off;` 디렉티브 설정 반영. [cite: 95]

#### [7-4] 취약한 보안설정 (일반)
* [cite_start]**상세 내용:** 응답 공통 헤더 보안 강화 상태 점검 결과 `Content-Security-Policy(CSP)` 헤더 및 보안 전송 강제화 헤더인 `Strict-Transport-Security(HSTS)` 지시어가 완전 누락된 상태입니다. [cite: 96]
* [cite_start]**대응 방안:** Nginx 환경 프록시 레이어에 필수 웹 보안 헤더 설정 추가 수립 및 배포. [cite: 96]

### 4.8 기타 취약점

#### [8-1] 비즈니스 로직 결함 - 재고 선점 결함 (-)
* **상세 내용:** 렌터카 예약 비즈니스 프로세스 결함. [cite_start]**사용자가 결제를 최종 완료하지 않고 단지 예약 요청 단계(soft-hold)에만 머물러도 인프라 DB 내 실재고 수량이 먼저 차감**됩니다. [cite: 101] [cite_start]악의적인 공격자가 여러 세션으로 결제 의사 없이 특정 차량/일정을 무한 선점해 둘 경우 실제 이용하려는 정상 사용자의 예약 진입이 완전 차단되는 **비즈니스적 서비스 거부(DoS) 사태 유발 가능**. [cite: 101]
* [cite_start]**대응 방안:** 결제 대기 상태 진입 시 타임아웃(TTL: 예: 10분 설정) 스케줄러 메커니즘을 부여하여 결제 미진행 시 선점 재고가 자동 반환(soft-release)되는 구조적 로직 보완 수립. [cite: 101]
